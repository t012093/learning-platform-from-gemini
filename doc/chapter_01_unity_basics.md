# 第1章｜Unity基礎（AI時代版・最重要）

## 1-1. Unityの世界観

### Scene / GameObject / Component
Unityの構造は「演劇」や「映画撮影」に例えると理解しやすいです。
*   **Scene (シーン)**: 場面。タイトル画面、ゲーム画面、リザルト画面など。一度に1つ（または複数）の舞台が存在します。
*   **GameObject (ゲームオブジェクト)**: 舞台上の役者や大道具。プレイヤー、敵、床、照明、カメラなど、シーンにあるものはすべてGameObjectです。
*   **Component (コンポーネント)**: 役者に与えられる「役割」や「台本」。GameObjectにComponentをつけることで、ただの箱が「物理挙動する箱」になったり、「光る箱」になったりします。

### Unityは「状態の集合体」
*   Unity開発とは、適切なGameObjectに適切なComponentをくっつけて、それらのパラメータ（状態）を調整していく作業です。
*   コード（Script）もComponentの一種です。

## 1-2. Scene設計の基礎

### 1シーン1役割
*   「全部入り」のシーンを作らないこと。管理が複雑になり、AIへの指示もしにくくなります。
*   「TitleScene」「GameScene」「ResultScene」のように役割を明確に分けます。

### 軽いシーン設計
*   シーンが重いとテストプレイの起動が遅くなり、バイブコーディングのリズムが崩れます。
*   必要なものだけを配置し、後から読み込む設計を意識します。

### なぜシーンを増やしすぎてはいけないか
*   逆に細切れにしすぎると、シーン間のデータの受け渡し（スコアやプレイヤーの状態など）が複雑になります。バランスが重要です。

## 1-3. GameObjectとComponent

### GameObjectは箱
*   作成直後のGameObjectは、位置情報しか持たないただの「空の箱」です。

### Componentが振る舞い
*   **Transform**: 位置・回転・大きさ（全てのGameObjectが持つ）。
*   **Mesh Renderer**: 形を表示する。
*   **Box Collider**: 当たり判定を持つ。
*   **Rigidbody**: 物理演算（重力など）に従う。
*   **Audio Source**: 音を鳴らす。
*   これらを組み合わせるパズル感覚を持ちましょう。

## 1-4. Transform（事故防止）

### Position / Rotation / Scale
*   **Position (位置)**: X, Y, Z座標。
*   **Rotation (回転)**: 向き。
*   **Scale (大きさ)**: サイズ。

### Local / Worldの違い
*   **World**: ゲーム世界全体から見た絶対的な値。
*   **Local**: 親オブジェクトから見た相対的な値。
*   AIに指示するとき、「ローカル座標で右に動かして」なのか「ワールド座標で」なのかを区別できないと、意図しない挙動になります。

### 親子関係の考え方
*   UnityではHierarchyウィンドウでドラッグ＆ドロップすることで、オブジェクトの親子関係を作れます。
*   親が動けば子も動く。戦車の砲塔（子）は戦車本体（親）についていく、といった表現に使います。
